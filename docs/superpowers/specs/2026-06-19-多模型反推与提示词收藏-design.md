# 多模型反推 + 提示词收藏 · 设计文档

- 日期：2026-06-19
- 范围：`wd14-tagger-web`（后端 FastAPI + 前端 Vue3/naive-ui）
- 参考实现：`tagger_prompt/scripts/{tagger_prompt.py, taggers_core.py}`（同仓库根目录的 Forge 扩展）

## 1. 背景与目标

当前反推功能硬编码只支持 WD14 一个模型；顶部页签只有「上传 / 图库 / 随机 / 设置」，缺少面向"已有提示词文本"的处理入口。本设计实现两个独立功能：

1. **多模型反推**：把参考实现 `tagger_prompt` 中的全部 7 个反推模型接入，前端可选；默认选上次使用的模型；已下载的模型在选择器中标绿显示「已下载」。
2. **提示词收藏（新页签）**：用户粘贴一段提示词文本，按当前分类词表（`tag_rules.yaml` + `quality_template.yaml`）拆分到各分类，可编辑、可上传多张示例图、可保存为收藏供下次复用；与图库物理隔离。

## 2. 现状摘要

- 反推链路：`deps.get_tagger()`（`@lru_cache` 单例）→ `WD14Tagger`（[backend/tagger/core.py](../../../backend/tagger/core.py)，只支持 wd14，首次反推自动下载）→ `tag_image(pil, gen_th, char_th, use_char)`。
- 调用点：`POST /api/images/{id}/tag`（[routes_images.py](../../../backend/api/routes_images.py)，`TagParams` 无 model 字段）；批量 `POST /api/batch/tag`（[routes_batch.py](../../../backend/api/routes_batch.py)）→ `tasks/queue.py` `_run`（无 model）。
- 分类引擎：`Classifier.classify(raw_tags)`（[classifier/engine.py](../../../backend/classifier/engine.py)）把 `{tag:score}` 归到 `quality/head/clothing/view/action/scene` + `extras`。其中 `quality` 无条件用模板全填；5 个内容类按 `priority` 顺序用 `exact/suffix/contains` 匹配；未匹配入 `extras`。
- 分类词表：[tag_rules.yaml](../../../backend/config/tag_rules.yaml)（5 类规则）+ [quality_template.yaml](../../../backend/config/quality_template.yaml)（质量词模板）。
- 顶部菜单：[App.vue](../../../frontend/src/App.vue) `n-menu` 4 项；路由 [router.ts](../../../frontend/src/router.ts)。
- 前端 prompt 拼接：`buildPrompt(meta)`（[detail-utils.ts](../../../frontend/src/detail-utils.ts)），顺序 `quality→head→clothing→view→action→scene`，与后端 `build_prompt` 一致。
- 数据模型：[models.py](../../../backend/models.py) `Meta.model` 字段已存在（默认 `"wd14"`），但此前未真正写入。

## 3. 决策记录（已与用户确认）

| 决策点 | 选择 |
|---|---|
| 未下载模型如何处理 | **先手动下载才可用**（独立「下载」按钮，下完才能选中反推） |
| 模型选择器形态 | **下拉菜单**，每项标注「已下载」(绿) / 「未下载」 |
| 提示词收藏示例图数量 | **多张** |
| 新页签命名 | **提示词收藏** |

## 4. 架构选型

- **后端 tagger 多模型化 → Spec 驱动的通用 ONNX tagger**。7 个模型里 wd14 / wd3 / wd_vit_v3 / wd_eva_v3 / wd_conv_v3 / e621 预处理完全相同（RGB→BGR、不归一化、读 `selected_tags.csv`、category 0/4/9 分桶），仅 DDB(DeepDanbooru) 不同（RGB、`/255.0` 归一化、读 `tags.txt`、无 category 分桶）。用一张 spec 表吸收这点差异，一个通用类按 spec 加载，避免 7 份几乎重复的类。
- **提示词收藏存储 → 独立 JSON 存储**。`data/promptbox/items.json` + `data/promptbox/images/<id>/`，全新 `PromptboxStore`，与图库 `data/images/` 物理隔离。

## 5. 需求1 · 多模型反推

### 5.1 后端

**新增 [backend/tagger/models_spec.py](../../../backend/tagger/models_spec.py)**：模型 spec 表（从 `tagger_prompt.py` 的 `_MODEL_DOWNLOADS` 移植，含下载源）。每条 spec：

| key | label | folder | 文件 | prep | tag 来源 |
|---|---|---|---|---|---|
| `wd14` | WD14 | `wd14` | model.onnx, selected_tags.csv | `bgr_wd` | csv |
| `wd3` | WD3 | `wd_swinv2_v3` | model.onnx, selected_tags.csv | `bgr_wd` | csv |
| `wd_vit_v3` | WD ViT v3 | `wd_vit_v3` | model.onnx, selected_tags.csv | `bgr_wd` | csv |
| `wd_eva_v3` | WD EVA v3 | `wd_eva_v3` | model.onnx, selected_tags.csv | `bgr_wd` | csv |
| `wd_conv_v3` | WD Conv v3 | `wd_conv_v3` | model.onnx, selected_tags.csv | `bgr_wd` | csv |
| `ddb` | DDB | `deepdanbooru` | model.onnx, tags.txt | `ddb` | txt |
| `e621` | E621 | `e621` | model.onnx, selected_tags.csv | `bgr_wd` | csv |

数据结构：`ModelSpec(key, label, folder, files: dict[name,url], prep: "bgr_wd"|"ddb", tag_source: "csv"|"txt")`，外加 `MODEL_SPECS: dict[str, ModelSpec]`、`DEFAULT_MODEL_KEY = "wd14"`、`is_downloaded(key) -> bool`（扫描 `MODELS_DIR/<folder>` 下 `files` 是否齐全）。

下载源沿用 `tagger_prompt`（HuggingFace `SmilingWolf/*`、`chinoll/deepdanbooru`、`silveroxides/Z3D-E621-Convnext`）。

**重构 [backend/tagger/core.py](../../../backend/tagger/core.py)**：`WD14Tagger` → `OnnxTagger(spec: ModelSpec)`。
- 保留方法签名 `ensure_loaded / _download / _load / _prep / tag_image(pil, gen_th=0.35, char_th=0.90, use_char=True) -> dict[str,float]`。
- `model_dir = MODELS_DIR / spec.folder`；`_download` 用 `spec.files` 下载；`_load` 按 `spec.tag_source` 读 csv（category 分桶）或 txt（无分桶，全部当 general）；`_prep` 按 `spec.prep`：`bgr_wd` = BGR 不归一化，`ddb` = RGB `/255.0`。
- `tag_image`：`bgr_wd` 走 general+char 双阈值；`ddb` 走单阈值（与 `taggers_core.DeepDanbooruTagger` 一致）。返回 `{display_tag: score}`，display_tag 把 `_` 替换为空格。
- 保留 `_download` 的"下载不完整校验 + `.part` 原子替换 + 失败清理"逻辑（来自现有 `WD14Tagger._download`）。

**改 [backend/deps.py](../../../backend/deps.py)**：去掉 `@lru_cache` 单例。
- `get_tagger(model_key: str = DEFAULT_MODEL_KEY) -> OnnxTagger`：用模块级 `dict[str, OnnxTagger]` 缓存实例，按 key 取/建。
- `get_classifier()` / `get_storage()` 保持不变。

**改 [backend/models.py](../../../backend/models.py)**：`Meta.model` 已存在，确认在反推/上传流程写入实际 key（见 5.2/5.3）。新增 `PromptboxItem`（见 6.1）。

## 6. 需求2 · 提示词收藏（新页签）

### 6.1 后端 · 独立存储

**新增 [backend/storage/promptbox_store.py](../../../backend/storage/promptbox_store.py)**：`PromptboxStore(data_root = settings.PROMPTBOX_DIR)`，风格仿 [store.py](../../../backend/storage/store.py)（防目录穿越、`new_id`、原子写）。

目录布局：
```
data/promptbox/
  items.json            # 收藏清单（原子写：写 .tmp 再 replace）
  images/<id>/          # 每条收藏的示例图（多张，原名保留扩展名）
```

`PromptboxItem`（pydantic，放 [models.py](../../../backend/models.py)）：
```python
class PromptboxItem(BaseModel):
    id: str
    title: str = ""
    raw_prompt: str                 # 用户最初粘贴的原文
    categories: dict[str, list[str]]  # {quality,head,clothing,view,action,scene}
    extras: list[str] = []
    image_names: list[str] = []     # 该 item 的示例图文件名
    created_at: str
    updated_at: str
```
> 存用户**处理过**的拆分结果（可在前端手改），不是原始反推；`raw_prompt` 单独保留原文以便下次重拆或溯源。

`PromptboxStore` 方法：`list_all() -> list[PromptboxItem]`、`get(id)`、`create(item, image_files) -> PromptboxItem`、`update(id, item, new_image_files, remove_image_names)`、`delete(id)`（连带 `images/<id>/`）、`image_path(id, name)`、`save_image(id, file) -> name`。读写均落 `items.json`（全量读写，条目量级小）。

### 6.2 后端 · 拆分逻辑（复用 Classifier）

**改 [classifier/engine.py](../../../backend/classifier/engine.py)**：新增 `split(text: str) -> dict[str, list[str]]`，与现有 `classify` 并列，**不改动 `classify`**。
- 文本按 `,` 与换行拆 tokens，`strip` 去空，保留原始大小写与出现顺序。
- 复用现有 `_match` + `self.priority` 把 token 归到 `head/clothing/view/action/scene`。
- `quality` **改为「按匹配」**：用 `self.quality_tags`（来自 `quality_template.yaml`）当 exact 词表参与匹配，只收提示词里实际出现的质量词（不再像 `classify` 那样无条件全填模板）。
- 未匹配 → `extras`。
- 返回 `{quality, head, clothing, view, action, scene, extras}`，每类是按出现顺序的 `list[str]`（无分数、无 phrase）。

### 6.3 API · 新增 [routes_promptbox.py](../../../backend/api/routes_promptbox.py)（prefix `/api/promptbox`）

- `POST /api/promptbox/split`  body=`{text: str}` → `{categories: {...}, extras: [...]}`（纯预览，不落库）。调 `get_classifier().split(text)`。
- `GET /api/promptbox` → `list[PromptboxItem]`（不含图，前端用 `image_names` 取）。
- `GET /api/promptbox/{id}` → 单条。
- `POST /api/promptbox`  multipart：表单 `title`、`raw_prompt`、`categories`(JSON)、`extras`(JSON)、`files`(多张图，可选) → 创建。
- `PUT /api/promptbox/{id}`  multipart：同上 + `remove_image_names`(JSON，可选) → 更新（增删图）。
- `DELETE /api/promptbox/{id}` → 删除（连带图目录）。
- `GET /api/promptbox/{id}/image/{name}` → `FileResponse`（防穿越）。

### 6.4 前端

**改 [App.vue](../../../frontend/src/App.vue)** 菜单加 `{ label: '提示词收藏', key: 'promptbox' }`；**改 [router.ts](../../../frontend/src/router.ts)** 加 `{ path: '/promptbox', component: PromptBoxPage }`。

**新增 [views/PromptBoxPage.vue](../../../frontend/src/views/PromptBoxPage.vue)**，上下两区：

**① 拆分工作区**
- `NInput` textarea 粘贴提示词 → 「拆分」按钮 → `POST /api/promptbox/split`。
- 结果按 `quality/head/clothing/view/action/scene` + `extras` 分栏，每栏 `NDynamicTags`（可增删）——即"自己处理过"的编辑能力。各栏维护本地 `categories` ref。
- 「复制完整 prompt」按钮：按 `quality→head→clothing→view→action→scene` 顺序拼接（本地工具函数，与 `buildPrompt` 同序）。
- 「另存为收藏」：`NInput` 标题 + `NUpload`（多张，`multiple`）→ 组装 `FormData`（`categories`/`extras` 序列化为 JSON 字段）→ `POST /api/promptbox`。成功后刷新列表。

**② 收藏列表**
- 卡片网格：标题、`raw_prompt` 摘要、示例图缩略图（多张小网格/`NImage` group 预览）、分类色块徽章。
- 卡片操作：`编辑`（把该条 `categories`/`raw_prompt` 载入回工作区，可重拆或手改后再 `PUT`）、`复制 prompt`、`删除`（确认弹窗）。
- 顶部搜索框：按 `title` / `raw_prompt` 子串过滤。

**改 [api/client.ts](../../../frontend/src/api/client.ts)**：加 `splitPrompt(text)`、`listPromptbox()`、`getPromptbox(id)`、`savePromptbox(formData)`、`updatePromptbox(id, formData)`、`deletePromptbox(id)`、`promptboxImageUrl(id, name)`。

### 6.5 数据流

```
粘贴文本 → POST /split → Classifier.split() → 分栏(NDynamicTags 可编辑)
  → [复制 prompt(本地拼接)] 或 [另存: 标题+多图 FormData → POST /promptbox → items.json + images/<id>/]
列表 → GET /promptbox → 卡片 → 编辑(载入回工作区) / 复制 / 删除
```

## 9. 非目标（YAGNI）

- 模型下载的 SSE 实时进度条（v1 同步 + 「下载中…」提示）。
- HF 镜像 `HF_ENDPOINT` 自动配置（预留 settings，不实现切换 UI）。
- 提示词收藏的导出/导入、多用户、富文本编辑。
- 反推阈值随模型记忆（仍用全局 gen/char 阈值）。
- 提示词的语义化智能拆分（仅按现有词表规则匹配，不做 NLP）。

## 10. 风险与注意

- **下载耗时/失败**：WD 系列 ~400MB，国内 HF 可能慢或超时；同步下载会让单次请求长时间 pending。前端须明确「下载中…」并允许失败重试；后端 `_download` 失败清理 `.part`。
- **DDB 差异**：`tags.txt`（无表头、无 category）+ RGB `/255.0`，`OnnxTagger` 必须按 `spec.tag_source`/`spec.prep` 分支，不能套用 wd 逻辑。
- **`get_tagger` 缓存**：换 model key 时各自缓存实例，内存占用随已用模型数增长（最多 7 个 ONNX session）。可接受。
- **`Meta.model` 兼容**：旧 meta.json 的 `model` 缺省为 `"wd14"`，读取兼容；反推时覆写为实际 key。
- **`items.json` 并发**：全量读写，单进程 FastAPI 足够；写时 `.tmp` + replace 防半写。
- **目录穿越**：`promptbox` 的 `id`/`name` 须做与 `store.py` 一致的校验。
- **`Classifier.split` 不污染 `classify`**：新方法独立，quality 的"按匹配"语义只用于拆词，反推仍走原 `classify`（quality 全填模板）。
